#  DataPulse — Data Analytics Platform

A **production-grade, full-stack analytics platform** demonstrating clean data engineering, a FastAPI backend, and an interactive Vanilla JS dashboard.

---

##  Architecture

```

                ┌─────────────────┐

                └───────┬─────────┘
                        │ analyze.py

                ┌─────────────────┐

```

##  Quick Start

### Prerequisites
- Python 3.11+
- pip

### 1. Install dependencies

```bash
# Root (pipeline + tests)
pip install -r requirements.txt

# Backend
pip install -r backend/requirements.txt
```

### 2. Run the data pipeline

```bash
# Step 1 — Clean raw data
python clean_data.py

# Step 2 — Compute analytics
python analyze.py
```

Both scripts write logs to `logs/` and save outputs to `data/processed/`.

### 3. Start the backend

```bash
# From project root
uvicorn backend.main:app --reload --host 0.0.0.0 --port 8000
```

API docs: http://localhost:8000/docs

### 4. Open the dashboard

```bash
# Serve via Python's built-in HTTP server (handles ES modules correctly)
python -m http.server 3000 --directory frontend/public
```

Then open: **http://localhost:3000**

>  Open via HTTP, not `file://` — ES module `import` requires an HTTP origin.

| Method | Endpoint | Description |
|--------|----------|-------------|

All responses follow the envelope:
```json

```

## Design Decisions

### Data Cleaning (`clean_data.py`)

| Decision | Rationale |
|----------|-----------|
| Two-pass date parsing | Vectorised `pd.to_datetime` first, then `dateutil` per-row only for NaT rows — balances speed with flexibility |
| Email regex over `email-validator` lib | No external dependency; covers structural validity adequately for analytics use-case |
| Dedup by sort + drop_duplicates | Single vectorised operation; no row-by-row loops |
| `CleaningStats` dataclass | Structured report enables downstream logging/alerting without mutating the DataFrame |

### Backend (`backend/`)

| Decision | Rationale |
|----------|-----------|
| Singleton `AnalyticsService` | One DataFrame loaded per process — avoids repeated disk I/O on every request |
| TTL + double-checked locking | Thread-safe cache that auto-refreshes without restart |
| Layered architecture (routes → services → utils) | Routes stay thin; business logic is testable in isolation |
| Pydantic v2 response models | Strict serialisation boundary; OpenAPI schema generated automatically |

### Frontend (`frontend/`)

| Decision | Rationale |
|----------|-----------|
| Vanilla JS + ES modules | Zero build toolchain; immediately readable; interview-explainable |
| Chart.js CDN | No npm install; works via `file://`-served HTTP |
| Lazy section init | Only initialises a section when the user first clicks it — reduces cold-load API calls |
| Debounced search | Avoids API hammering on keystroke; 350ms delay matches perceived responsiveness |

##  Assumptions

1. `customers.csv` and `orders.csv` headers match the expected column names.
2. Churn window is **90 days** from the dataset's maximum `order_date` (configurable via `CHURN_DAYS` env var).
3. Only **completed** orders count toward revenue and customer spend.
4. The backend is run from the **project root** so `data/processed/` resolves correctly.
5. The frontend is served via HTTP (not `file://`) to allow ES module imports.

| Variable | Default | Description |
|----------|---------|-------------|
| `PROCESSED_DIR` | `data/processed` | Path to analytics CSVs |
| `CACHE_TTL_SECONDS` | `300` | DataFrame cache TTL in seconds |
| `CHURN_DAYS` | `90` | Days without order = churned |
| `CORS_ORIGINS` | `["*"]` | Allowed CORS origins |
